# 数据撤回 SOP V1

> 阶段定位：本 SOP 定义宠物主人撤回数据使用同意后的前台、数据、技术和临床处理流程。  
> 本阶段只做文档，不新增数据库、不新增 API。

## 1. 适用范围

适用于以下数据撤回请求：

```txt
客户不同意数据进入 AI 训练池
客户撤回既往同意
客户要求后续不再导出其病例数据
客户要求影像 / 检验 / 病历文本从训练候选集中排除
旧病例迁移时发现客户 opt-out
EMR webhook 入站时发现客户 opt-out
```

## 2. 角色分工

| 角色 | 职责 |
|---|---|
| 前台 / 客服 | 接收撤回请求，核对主人与宠物信息，填写撤回登记 |
| 医生 / 科室负责人 | 判断是否影响临床病历保存与诊疗流程 |
| 数据负责人 | 将病例 / 主人 / token 加入 opt-out registry |
| 技术负责人 | 更新训练池、导出集、Webhook、旧病例导入过滤规则 |
| 合规负责人 | 定期抽查撤回处理是否完成 |

## 3. 标准流程

### Step 1：接收请求

可接受渠道：

```txt
前台纸质表
电话登记
微信 / 在线表单
电子入院表单撤回入口
```

前台记录：

```txt
主人姓名
主人电话
宠物姓名
病历编号 / case_id
请求类型：不同意 / 撤回 / 仅排除训练 / 仅排除外部共享
请求时间
经办人
```

### Step 2：身份核对

最低核对：

```txt
主人姓名 + 电话
宠物姓名
最近一次就诊时间或病历编号
```

无法核实时：

```txt
登记为 pending_verification
不得立即删除或改动临床病历
由数据负责人二次确认
```

### Step 3：登记 opt-out registry

记录内容见：

```txt
docs/compliance/OPT_OUT_REGISTRY_SPEC.md
```

状态建议：

```txt
received
pending_verification
active
completed
rejected
superseded
```

### Step 4：技术处理

按数据所在位置处理：

```txt
训练候选池：标记不可用
待标注任务：移出任务池
导出数据集：从后续导出中排除
旧病例导入 CSV：过滤该 case_id / patient_token
EMR Webhook：入站后匹配 opt-out，不进入训练用途
生产病历：依法保留，不因训练撤回而删除临床病历
```

### Step 5：时间要求

```txt
2 个工作日内：确认收到请求
30 日内：停止新数据进入训练或导出
90 日内：完成可移除数据的不可用标记
每季度：抽查撤回登记与实际排除结果
```

## 4. 不可删除事项

以下内容通常不因 AI 数据撤回而删除：

```txt
依法需要保留的临床病历
收费 / 财务记录
医疗纠纷或监管要求保留的记录
已生成且无法按个体回溯删除的模型权重
不可变审计日志
```

处理原则：

```txt
临床记录继续保留
训练 / 导出 / 再标注用途停止
后续数据管道必须排除
```

## 5. 异常处理

### 5.1 客户要求删除全部病历

处理：

```txt
转交医院负责人和合规负责人
区分“临床病历保存义务”和“AI / 研究用途撤回”
不得由前台直接删除病例
```

### 5.2 已经导出给外部合作方

处理：

```txt
记录导出批次
通知合作方停止后续使用
如合同约定支持删除或隔离，按合同执行
将处理结果写入撤回登记
```

### 5.3 多宠物同一主人

处理：

```txt
确认撤回范围：
仅某一只宠物
该主人所有宠物
仅某一就诊记录
全部历史数据
```

## 6. 验收标准

本 SOP 落地后，至少满足：

```txt
前台知道如何登记撤回
技术知道如何把 case_id / token 排除出训练数据
医生知道撤回不等于删除临床病历
合规负责人能抽查撤回登记
后续系统能根据 opt-out registry 做自动过滤
```
